## 5. Technical Risks and Fastest MVP Architecture

### Biggest Technical Risks (Ranked)

**Risk 1: Spreadsheet Grid Performance at Scale (HIGH)**
- The core UX promise is "works with 10,000+ products without slowing down"
- Loading 10,000 product rows with variations into a browser grid will choke unless you implement virtual scrolling (only render visible rows)
- AG Grid Community Edition (MIT license, free for commercial use) handles this natively with row virtualization
- Handsontable requires a paid commercial license -- ruled out
- **Mitigation:** Use AG Grid Community. Load products via paginated REST API calls (50-100 per request). Virtual scroll handles rendering. Server-side filtering/sorting for large catalogs.

**Risk 2: WooCommerce CRUD API Performance (MEDIUM)**
- The WooCommerce CRUD API (wc_get_products, WC_Product methods) is slow for bulk reads -- it loads full product objects with all meta
- Direct WP_Query with targeted meta queries is 5-10x faster but bypasses WooCommerce's data layer
- HPOS adds complexity -- products can be stored in traditional post meta OR the new orders table format
- **Mitigation:** Use WP_Query for reads (fast) with HPOS-aware meta queries. Use WooCommerce CRUD API for writes (safe, triggers hooks, maintains data integrity). Cache product data in a transient or custom table for grid rendering.

**Risk 3: Variation Editing Complexity (MEDIUM)**
- A product with 3 attributes and 5 values each = 125 possible variations
- Rendering parent + child rows in a spreadsheet grid requires careful UX (expand/collapse, indentation, parent vs variation field differences)
- Bulk-editing variation prices while keeping parent product in sync requires WooCommerce's _price meta recalculation
- **Mitigation:** AG Grid supports master-detail (expandable rows) natively. On save, call WC_Product_Variable::sync() to recalculate parent price ranges.

**Risk 4: WordPress.org Plugin Review Approval (LOW but blocking)**
- WordPress.org plugin team reviews every new submission; takes 2-4 weeks
- Common rejection reasons: external API calls without disclosure, minified JS without source, unclear licensing, PHPUnit test failures
- AG Grid Community is MIT-licensed and can be bundled
- **Mitigation:** Follow WordPress coding standards from day one. Include unminified source. No external API calls in free version. Submit early (week 4) to parallelize review with premium feature development.

**Risk 5: Data Corruption Liability (LOW but reputation-killing)**
- BEAR's worst reviews are about data loss. One bad data loss incident early on could kill the plugin's reputation permanently.
- **Mitigation:** Before every bulk save, serialize affected product data to a JSON snapshot in wp_options (or a custom table). The undo feature reads from this snapshot. Belt-and-suspenders: warn users in UI before large operations (50+ products), suggest WooCommerce export as backup.

### Fastest MVP Architecture

```
Plugin Structure:
amazingplugins-bulk-editor/
  amazingplugins-bulk-editor.php    -- Plugin bootstrap, hooks registration
  includes/
    class-product-query.php         -- HPOS-aware product fetching (WP_Query wrapper)
    class-product-saver.php         -- WooCommerce CRUD save operations + undo snapshots
    class-rest-api.php              -- Custom REST endpoints for grid data
    class-bulk-operations.php       -- Price set/adjust, text search-replace, category change
  assets/
    js/
      ag-grid-community.min.js     -- AG Grid Community (MIT, ~300KB gzipped)
      bulk-editor.js               -- Grid initialization, cell editors, keyboard nav
    css/
      bulk-editor.css              -- Grid styling, WP Admin integration
  readme.txt                        -- WordPress.org listing (this IS marketing copy)

Tech Stack:
  Backend: PHP 8.1+, WordPress REST API, WooCommerce CRUD API
  Frontend: AG Grid Community Edition (MIT), vanilla JS (no React/Vue -- keeps bundle small)
  Data flow: REST API -> JSON -> AG Grid -> REST API -> WC CRUD save
  Undo: JSON snapshots in wp_options (capped at last 5 operations, auto-cleanup via cron)
```
